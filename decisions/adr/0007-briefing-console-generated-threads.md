# ADR-0007 — Briefing console: LLM-generated decision threads, reuse of the handoff write path

- **Status:** Accepted
- **Date:** 2026-06-22
- **Deciders:** Yuri (Jim Green), code-claude
- **Builds on:** ADR-0003 (local-first synthesis), ADR-0005 (handoff emission), ADR-0006 (grounded chat)

## Context

The cockpit redesign ("OJF Editorial" command deck) adds a centrepiece **Briefing** console: a
Chief-of-Staff surface where the morning's most decision-worthy beads are turned into seeded
threads, each resolving through a decision tree into an approvable **Handoff Artifact**. Unlike the
existing pods, the Briefing's threads are NOT a presentation of read-model rows — a decision tree
(catch-up → question → recommended action → acceptance criteria) is *authored/generated content*.
The question was where that content comes from and how its single real action (emit a brief) relates
to the chat's existing write path.

## Decision

1. **Threads are generated by a cockpit-server Chief-of-Staff pass, with a deterministic floor.**
   `GET /api/briefing` grounds the local model (Ollama `qwen2.5:7b`, ADR-0006 discipline) on the
   `CockpitSnapshot` and asks for up to 4 threads, each with ONE recommended action. To keep a 7B
   model reliable the model emits a flat per-thread action (not a nested branch tree); the server
   attaches a deterministic `defer` branch. Every recommended artifact is gated by the existing
   `validateBriefDraft` (repo must exist, criteria present) — threads whose emit could never write
   are dropped. If nothing survives, or the model is unavailable/returns junk, the endpoint returns
   `briefingFallback(snapshot)` (`source:'deterministic'`) — honest threads built straight from real
   lane data. Ollama-ONLY, no cloud cascade. The `source` flag (`llm` | `deterministic`) is shown.

2. **The generator only proposes; every emit stays human-approved (reuses ADR-0005).** A deliver
   branch's "Approve & emit" maps the `BriefingArtifact` onto the existing `BriefCandidate`/`BriefBody`
   shape (`artifactToCandidate`) and writes through the SAME gated path as the chat
   (`emitArtifact` → `approveDraft`: validate → path-guard → write `<repo>/.handoff/`, never
   overwrite, never create a repo). `POST /api/briefing/emit`. There is **no second write path** —
   the explicit Approve click in the decision tree is the ADR-0005 per-emission human gate.

3. **Coexists with Cockpit Chat (ADR-0006), does not replace it.** The chat remains the general,
   open-line rail; the Briefing is the bead-specific console. Both share the one upstream write.

4. **The non-handoff verbs are deferred.** `defer`/`archive` branches (→ `bead-snooze`/`bead-archive`)
   and the repo-scaffold branch render with an honest "verb not built — coordination-design §6"
   affordance. `emitIntent` stays the single seam so those verbs slot in when core ships them
   (ADR-0002 / coordination-design §6: `emitEvent`/`bead_events`, `queue-*`, scaffold).

## Consequences

- The Briefing is real from day one without waiting on the coordination layer, and it never lies:
  generated content is gated + labelled, and only the one proven write verb is live.
- A 7B model's threads are terse and occasionally thin; richer multi-branch trees and prompt tuning
  are future work. The deterministic fallback guarantees the console is always populated and honest.
- UI state (active thread / chosen / approved branch) is local + persisted (`mc.cockpit.v1`); domain
  content always comes from `/api/briefing`, never hard-coded past the typed loading placeholder.
